### Journey Builder Timing Classifier

**Status: Ôťů PASS (14/14) ÔÇö with one spec divergence flagged**

| Scenario | Expected lag class | Actual | Status |
|---|---|---|---|
| `immediate` duration | `immediate`, no platform risk | Correct | Ôťů |
| `one_to_seven_days` duration | `short_lag`, Meta marginal | Correct | Ôťů |
| `one_to_four_weeks` duration | `long_lag`, Meta outside window | Correct | Ôťů |
| `over_one_month` duration | `deep_lag`, critical | Correct | Ôťů |
| Non-conversion event (`page_view`) | No classification applied | `CONVERSION_ACTION_KEYS` check skips it | Ôťů |
| Google risk at `one_to_four_weeks` | Marginal (not outside window) | `marginal` returned | Ôťů |
| Google risk at `over_one_month` | Marginal (up to 90 days) | `marginal` returned | Ôťů |
| `LONG_LAG_EVENT_TYPES` (`generate_lead`) | No runtime effect | Reserved, unused | Ôťů |

**FUNC-03 (P2) ÔÇö Timing lag-class bands broader than PRD spec:**

The PRD defines `short_lag` as "same-session or same-day (within 24h)" and `long_lag` as "1ÔÇô7 days". The implementation maps:
- `one_to_seven_days` Ôćĺ `short_lag` (Meta marginal)
- `one_to_four_weeks` Ôćĺ `long_lag` (Meta outside window)

A 5-day lag therefore classifies as `short_lag` / Meta-marginal, not `long_lag` / outside-window. The implementation effectively compresses the `long_lag` band out of existence. A 5-day lag on a Meta campaign is well outside the 24h attribution window and should carry a higher risk flag.

**Impact:** Customers with 1ÔÇô7 day conversion cycles are shown an optimistic "marginal" Meta risk instead of "outside window". This could lead to mis-timed Meta bidding configuration.

**Recommended fix:** Requires a product decision on whether to add a `one_to_seven_days` bucket with `outside_window` Meta risk or redefine the band boundaries. No code change without approval.

### Andromeda Readiness Score

**Status: Ôťů PASS (15/15) ÔÇö with one duplicate-criterion bug flagged**

**Scoring formula (sums to 100):**

| Criterion | Points | DB condition |
|---|---|---|
| `consent_configured` | +20 | `consent_configs` row exists |
| `server_side_enabled` | +20 | any active CAPI provider |
| `capi_connected` | +20 | any active CAPI provider |
| `click_id_capture` | +15 | `planning_outputs` row with `output_type='gtm_container'` |
| `enhanced_conversions` | +15 | active provider has email/phone in `identifier_config.enabled_identifiers` |
| `health_score_strong` | +10 | `health_scores.overall_score > 80` |

**FUNC-01 (P2) ÔÇö `server_side_enabled` and `capi_connected` are computed identically:**

Both criteria query `activeProviders.length > 0` (active CAPI provider count). An org with exactly one active CAPI provider and nothing else therefore scores **40 points** (20 + 20) rather than 20. The score is over-rewarding CAPI presence.

**Impact:** A new customer who has connected CAPI but not configured consent, click-ID capture, enhanced conversions, or achieved a strong health score will see a readiness score of 40 instead of the expected 20. The score is inflated, potentially reducing urgency to complete setup.

**Recommended fix:** Differentiate the two criteria. Options:
- `server_side_enabled` = any CAPI provider exists (any status), `capi_connected` = active provider with at least one successful delivery
- Or collapse into a single 20-point CAPI criterion and redistribute the freed 20 points to a genuinely distinct dimension (e.g. signal freshness from Phase 2 DEV-03)

No code change without product sign-off on which definition is intended.

### Strategy Gate Behaviour

**Status: Ôťů PASS ÔÇö with one stale-label finding**

The `strategyGate` middleware blocks `POST /api/planning/sessions`, `POST /api/journeys`, and `POST /api/:orgId/clients/:clientId/deploy` unless the org has at least one strategy brief (any status ÔÇö not required to be locked). The frontend banner is informational; enforcement is backend-only.

**FUNC-02 (P2) ÔÇö StrategyGateBanner displays stale verdict label field names:**

`StrategyGateBanner.tsx` references verdict values `keep`, `add_proxy`, `switch` when rendering the brief status. These were the original field values, later renamed to `CONFIRM`, `AUGMENT`, `REPLACE` (Phase 2 DEV per migration `20260421`). The banner's verdict display will always fall through to the default label `'Strategy defined'` for any current brief, because no brief will have the old field values.

**Impact:** Low ÔÇö the banner still shows a green "Conversion event locked" state when a brief exists; the verdict text is just wrong. No workflow is blocked.

**Recommended fix:** Update `StrategyGateBanner.tsx` to use `CONFIRM | AUGMENT | REPLACE` label mapping.
